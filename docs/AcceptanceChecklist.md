## 0. 驗收前準備

- [ ] 後端已啟動（`mvn spring-boot:run`），終端機無錯誤訊息
- [ ] 前端已啟動（`npm run dev`），瀏覽器開啟對應網址後畫面顯示「● 後端已連線」
- [ ] 樓層圖顯示 1F 閱覽 A 區的所有座位，且初始狀態多數為「空閒」（綠色）

## FR-01 查詢座位狀態

**需求**：系統應於 2 秒內以視覺化樓層圖呈現所有座位的即時狀態（空閒/使用中/暫離/已預約）。

**操作步驟**：
1. 開啟前端首頁，記下任一座位（例如 A1）目前的顏色與狀態文字。
2. 不要點擊該座位，改用另一個操作（例如對另一張座位執行「預約」），使該座位狀態改變。
3. 等待最多 2 秒，**不要手動刷新頁面**，觀察樓層圖與統計卡片。

**預期結果**：該座位的顏色與狀態文字在 2 秒內自動更新為新狀態，頂部統計卡片（空閒/已預約/使用中/暫離 人數）同步變化。

**通過條件**：
- [ ] 不需手動刷新頁面，座位狀態在 ≤2 秒內自動反映後端最新狀態
- [ ] 樓層圖的顏色與文字標籤（空閒/已預約/使用中/暫離）與圖例一致
- [ ] 頂部統計卡片數字正確反映目前所有座位的狀態分布

## FR-02 QR Code Check-in

**需求**：讀者掃描座位 QR Code 後，系統切換座位狀態為「使用中」（本版為簡化版，QR 為真實編碼，掃描提交以「模擬掃描成功」按鈕觸發）。

**操作步驟**：
1. 點選一個「空閒」狀態的座位（例如 A1）。
2. 在操作面板點擊「掃 QR Check-in」。
3. （可選）使用手機 QR Code 掃描 App 掃描畫面上出現的 QR Code。
4. 點擊「模擬掃描成功」。

**預期結果**：
- 步驟 2 後出現 QR Code 卡片。
- 步驟 3（若執行）掃描結果應為字串 `SLSMS:CHECKIN:A1`（依所選座位編號）。
- 步驟 4 後座位狀態變為「使用中」，並顯示「使用者 demo-user-001」。

**通過條件**：
- [ ] 點擊「掃 QR Check-in」後立即顯示 QR Code 圖卡
- [ ] （可選）手機掃描該 QR Code 可解析出 `SLSMS:CHECKIN:{座位編號}` 字串
- [ ] 點擊「模擬掃描成功」後，座位狀態變為「使用中」，且顯示「使用者 demo-user-001」
- [ ] 樓層圖上該座位顏色同步變為「使用中」對應顏色

## FR-03 即時預約

**需求**：讀者可選擇空閒座位進行即時預約；預約成功後 15 分鐘內未 Check-in 則自動釋放（自動排程為 V2 規劃中，本版不示範自動觸發）。

**操作步驟**：
1. 點選一個「空閒」狀態的座位。
2. 在操作面板點擊「預約」。

**預期結果**：座位狀態變為「已預約」，操作面板顯示「預約者 demo-user-001」。

**通過條件**：
- [ ] 點擊「預約」後，該座位狀態變為「已預約」
- [ ] 操作面板顯示「預約者 demo-user-001」
- [ ] 樓層圖上該座位顏色同步變為「已預約」對應顏色
- [ ] （記錄用，不勾選即視為已知限制）15 分鐘逾時自動釋放為 V2 規劃中，本版不示範自動觸發

## FR-05 暫離模式

**需求**：使用中讀者可將座位切為「暫時離開」最多 30 分鐘；逾時則自動釋放（自動排程為 V2 規劃中）。本版另外實作了「僅本人可操作」的歸屬驗證（見 §10）。

**操作步驟**：
1. 點選一個「使用中」狀態的座位（若無，先依 FR-02 完成一次 Check-in）。
2. 在操作面板點擊「暫時離開」。
3. 觀察座位狀態變為「暫離」後，再點擊「回來」。

**預期結果**：
- 步驟 2 後座位狀態變為「暫離」。
- 步驟 3 後座位狀態變回「使用中」，且「使用者 demo-user-001」保持不變。

**通過條件**：
- [ ] 點擊「暫時離開」後，該座位狀態變為「暫離」
- [ ] 點擊「回來」後，該座位狀態變回「使用中」，使用者資訊不變
- [ ] 樓層圖上該座位顏色隨狀態切換同步更新
- [ ] （記錄用，不勾選即視為已知限制）30 分鐘逾時自動釋放為 V2 規劃中，本版不示範自動觸發

## FR-06 噪音/溫度警示

**需求**：感測器讀數逾門檻值時，系統應推播警示給該區管理員與數位看板。Version 2.0 將溫度新增為第二種被監測的環境指標，沿用與噪音相同的 Observer Pattern 架構。

### FR-06a 噪音警示

**操作步驟**：
1. 在頁面下方找到「A 區噪音感測器」面板，記下目前的門檻值（預設 65 dB）。
2. 重複點擊「+20 dB（突發）」，直到顯示的 dB 數值超過門檻。
3. 觀察畫面是否出現警示橫幅，並查看「近期警示」清單。
4. 點擊「重置」將噪音調回門檻以下，再觀察警示橫幅是否消失。

**預期結果**：
- 噪音超過門檻後，出現帶有脈動效果（pulsing）的橫幅，文字包含「Observer 觸發」與數位看板訊息。
- 「近期警示」清單新增一筆紀錄，時間為觸發當下。
- 噪音降回門檻以下後，警示橫幅消失（邊緣觸發：不會持續重複新增警示）。

**通過條件**：
- [ ] 噪音數值超過門檻後，出現脈動警示橫幅，文字含「Observer 觸發」
- [ ] 「近期警示」清單新增一筆對應時間的紀錄
- [ ] 將噪音調回門檻以下後，警示橫幅消失
- [ ] 再次將噪音調至超過門檻，「近期警示」會新增**新一筆**紀錄（而非重複顯示舊紀錄），驗證邊緣觸發行為

### FR-06b 溫度警示（Version 2.0 新增）

**操作步驟**：
1. 在頁面下方找到「A 區溫度感測器」面板，記下目前的範圍（預設 18–28°C）。
2. 點擊「+10°C（過熱）」數次，直到顯示的溫度超過上限。
3. 觀察畫面是否出現警示橫幅，並查看「近期警示」清單。
4. 點擊「重置」將溫度調回範圍內，再觀察警示橫幅是否消失。
5. 點擊「−10°C（過冷）」數次，直到顯示的溫度低於下限，重複觀察警示橫幅與「近期警示」清單。

**預期結果**：
- 溫度超過上限後，出現帶有脈動效果（pulsing）的橫幅，文字包含「Observer 觸發」與數位看板訊息（過熱）。
- 溫度低於下限後，同樣出現警示橫幅，但文字為「過冷」對應訊息。
- 「近期警示」清單新增對應紀錄,`type` 為 `TEMP_HIGH` 或 `TEMP_LOW`，與噪音的 `NOISE_HIGH` 各自獨立。
- 溫度回到範圍內後，警示橫幅消失（邊緣觸發：不會持續重複新增警示）。

**通過條件**：
- [ ] 溫度超過上限後，出現脈動警示橫幅，文字含「Observer 觸發」與「過熱」相關訊息
- [ ] 溫度低於下限後，出現脈動警示橫幅，文字含「過冷」相關訊息
- [ ] 「近期警示」清單新增對應紀錄，且與噪音警示的紀錄互不干擾（各自獨立顯示）
- [ ] 將溫度調回範圍內後，警示橫幅消失
- [ ] 再次超出範圍（同方向）時，「近期警示」會新增**新一筆**紀錄，驗證邊緣觸發行為

## FR-08 使用分析報表

**需求**：系統提供使用率/環境警示熱點報表，以 Strategy Pattern (`ReportStrategy`) 實作，可擴充更多報表型態。Version 2.0 提供「座位使用率報表」與「環境警示熱點報表」兩種策略（資料來源為現有記憶體狀態，每日排程產出與 CSV 匯出仍為後續規劃）。

**操作步驟**：
1. 在頁面下方找到「使用率報表」面板。
2. 於下拉選單選擇「座位使用率報表」，點擊「重新整理」。
3. 觀察表格內容：整體使用率、各分區座位數與各狀態（空閒/已預約/使用中/暫離）數量、各分區使用率。
4. 於下拉選單切換為「環境警示熱點報表（噪音/溫度）」，點擊「重新整理」。
5. 觀察表格內容：各分區最新噪音/溫度讀數、門檻/範圍、警示狀態與近期警示次數。
6. （可選）先依 FR-06 觸發一次噪音或溫度警示，再重新整理本報表，確認對應分區的「近期警示」次數增加。

**預期結果**：
- 「座位使用率報表」表格列出每個分區的座位數、各狀態數量與使用率（百分比），總計與樓層圖目前狀態一致。
- 「環境警示熱點報表」表格列出每個分區的噪音/溫度讀數與警示狀態；觸發警示後對應分區的「近期警示」次數增加。
- 切換報表類型不需重新整理頁面，表格內容即時更新為對應報表的欄位。

**通過條件**：
- [ ] 下拉選單可選擇至少 2 種報表類型（座位使用率報表、環境警示熱點報表）
- [ ] 「座位使用率報表」的座位數量、各狀態統計與目前樓層圖狀態一致
- [ ] 「環境警示熱點報表」的噪音/溫度讀數與 FR-06 面板顯示的數值一致
- [ ] 觸發一次噪音或溫度警示後，重新整理「環境警示熱點報表」，對應分區「近期警示」次數較觸發前增加
- [ ] 對 `GET /api/reports/unknown-type` 呼叫，回應 HTTP 400（驗證未知報表型態的錯誤處理）

## 10. 設計模式可觀察驗收點

本節對應 SDD 第 3 節的兩個核心設計模式，驗收重點是「行為可被外部觀察到」，不需閱讀程式碼。

### 10.1 State Pattern — 非法操作回 409 並顯示紅色錯誤條

**驗收方式 A（API 層，使用 curl/Postman）**：

1. 確認某座位（例如 A1）目前狀態為「使用中」（先依 FR-02 完成 Check-in）。
2. 對該座位送出一個在「使用中」狀態下不允許的動作，例如再次 Check-in：
   ```
   curl -i -X POST http://localhost:8080/api/seats/A1/checkin -H "Content-Type: application/json" -d "{\"userId\":\"demo-user-001\"}"
   ```
3. 觀察回應的 HTTP 狀態碼與訊息。

**預期結果**：回應 HTTP **409 Conflict**，JSON 內容包含 `"status":409` 與類似 `"Action 'checkIn' is not allowed in state 'OCCUPIED'"` 的訊息。

**通過條件**：
- [ ] 對「使用中」座位重複呼叫 `/checkin`，回應為 **409 Conflict**
- [ ] 回應 JSON 包含 `status`、`error`、`message` 欄位，訊息清楚指出目前狀態與不允許的動作

**驗收方式 B（前端 UI，顯示紅色錯誤條）**：

1. 用兩個瀏覽器視窗（或一般視窗 + 無痕視窗）開啟同一個前端網址。
2. 兩個視窗都點選同一個「空閒」座位（例如 A2）。
3. 在視窗一點擊「預約」，**1 秒內**立刻切換到視窗二並同樣點擊「預約」（在視窗二的畫面尚未自動刷新為「已預約」之前點擊）。

**預期結果**：視窗二的請求會因座位已被預約而被拒絕，畫面上方出現**紅色背景的錯誤訊息條**，內容為後端回傳的錯誤訊息（如「Action 'reserve' is not allowed in state 'RESERVED'」）。

**通過條件**：
- [ ] 較慢送出請求的視窗，畫面出現紅色背景錯誤訊息條
- [ ] 錯誤訊息條文字為後端回傳的 409 錯誤訊息（非前端通用錯誤）
- [ ] 該視窗的座位狀態最終仍與後端一致（顯示「已預約」，而非誤判為操作成功）

> 補充：本版另外針對 `release` / `leaveTemp` / `comeBack` 加入「僅限座位目前使用者本人操作」的歸屬驗證（對應 SDD §4 的使用者歸屬驗證說明），同樣會回傳 409。若要驗收此項，需以 curl/Postman 對同一座位以不同 `userId` 呼叫上述端點，預期同樣回應 409 Conflict。

### 10.2 Observer Pattern — 噪音跨過門檻時警示條脈動且近期警示新增一筆

本項與 FR-06 操作相同，於此重複列出作為設計模式驗收的獨立勾選項：

**操作步驟**：在「A 區噪音感測器」面板，將噪音調整至超過門檻值。

**預期結果**：
- 出現脈動（pulsing）效果的警示橫幅。
- 「近期警示」清單即時新增一筆紀錄。

**通過條件**：
- [ ] 噪音超過門檻時，警示橫幅以脈動動畫呈現（非靜態樣式）
- [ ] 「近期警示」清單筆數較觸發前增加 1 筆，且時間戳記為觸發當下時間

### 10.3 Observer Pattern 擴充 — 同一套架構同時監測噪音與溫度（Version 2.0 新增）

本項驗證「不新增 Observer 架構、僅新增 Subject 即可擴充新感測器」的開放封閉原則。

**操作步驟**：
1. 同時將「A 區噪音感測器」調至超過門檻，並將「A 區溫度感測器」調至超過上限（或低於下限）。
2. 觀察兩個面板是否都各自出現脈動警示橫幅。
3. 查看「近期警示」清單。

**預期結果**：
- 噪音與溫度的警示橫幅各自獨立顯示，互不影響。
- 「近期警示」清單中同時出現 `type` 為 `NOISE_HIGH` 與 `TEMP_HIGH`（或 `TEMP_LOW`）的紀錄。

**通過條件**：
- [ ] 噪音與溫度警示可同時觸發，且各自的警示橫幅獨立顯示
- [ ] 「近期警示」清單同時包含噪音與溫度的警示紀錄
- [ ] 任一感測器恢復正常範圍，僅該感測器的警示橫幅消失，另一感測器不受影響

### 10.4 Strategy Pattern — 報表策略可替換（Version 2.0 新增）

**操作步驟**：
1. 在「使用率報表」面板的下拉選單，依序切換「座位使用率報表」與「環境警示熱點報表」。
2. 觀察每次切換後表格欄位與內容是否隨報表類型改變。

**預期結果**：
